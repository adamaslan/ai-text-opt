> **Free tier fit check**
> - 2 collections max тЖТ use 1 (`trader_qa`) + 1 reserved
> - 1,000,000 vectors ├Ч 768 dimensions тЖТ ~950 chunks fits with massive headroom
> - 1 full platform user (unlimited invited users for querying)

> - SDKs: Python (`pymilvus`), Node.js (`@zilliz/milvus2-sdk-node`), REST тАФ all supported

## Overview

A Retrieval-Augmented Generation pipeline that lets users query the Trader Q&A knowledge base through a Next.js chatbot. User questions are embedded, matched against chunked trader content in Zilliz Cloud, and passed to an LLM to produce grounded, context-aware responses from two trader perspectives.

## 2. Data Ingestion & Vectorization

### 2.1 Chunking Strategy

Split each markdown file by Q&A pairs using the existing document structure:

- **Primary splitter:** `**Q\d+.**` regex тАФ each question + both trader answers = one chunk
- **Secondary splitter:** `---` section dividers for outline/profile docs
- **Chunk size:** ~400 tokens, 50-token overlap
- **Minimum chunk length:** 100 tokens (discard headers-only fragments)

Each chunk carries metadata:
```json
{
  "source_file": "doc-1-options-and-swing-trading.md",
  "theme_number": 1,
  "theme_name": "Options Basics: Calls, Puts, and the Greeks",
  "doc_section": "Options & Swing Trading",
  "trader_1_present": true,
  "trader_2_present": true,
  "question_text": "What does it actually mean when a call option is 'in the money'?"
}
```

### 2.2 Embedding Model

Use **Zilliz's built-in embedding pipeline** тАФ no external embedding API call or key needed at ingestion time. Zilliz handles vectorization server-side when you insert text.

- **Built-in model:** `BAAI/bge-base-en-v1.5` (via Zilliz's embedding service) тАФ **768 dimensions**
- **Fits free tier:** 1,000,000 ├Ч 768-dim vectors; ~950 chunks uses <0.1% of capacity
- **Alternative (self-managed):** Call OpenAI `text-embedding-3-small` (1536D) or Hugging Face `all-MiniLM-L6-v2` (384D) before insert if you want model portability
- **Query time:** Zilliz also embeds the user query server-side via the same built-in pipeline тАФ no client-side embed step needed

> **Why built-in?** Eliminates a separate API dependency, reduces ingestion code, and keeps embedding model and index perfectly aligned. Switch to a self-managed model only if you need a specific embedding not in Zilliz's catalog.

### 2.3 Zilliz Cloud Collection Schema

**Collection name:** `trader_qa` (1 of 2 free-tier collections)

| Field | Type | Description |
|---|---|---|
| `id` | INT64 (primary key, auto) | Unique chunk identifier |
| `text_content` | VARCHAR(4096) | Full chunk text passed to built-in embedder |
| `question_text` | VARCHAR(512) | Extracted question for display in sources panel |
| `source_file` | VARCHAR(128) | Origin markdown filename |
| `theme_number` | INT64 | Theme 1тАУ50 (0 for non-Q&A docs) |
| `theme_name` | VARCHAR(256) | Human-readable theme label |
| `doc_section` | VARCHAR(128) | e.g. "Options & Swing Trading", "Industries & Sectors" |
| `embedding` | FLOAT_VECTOR(768) | Auto-generated by Zilliz built-in pipeline from `text_content` |

**Index:** AUTOINDEX on `embedding` (Zilliz manages HNSW parameters automatically on free tier тАФ no manual `M`/`efConstruction` config required)  
**Metric:** COSINE

### 2.4 Ingestion Script

`scripts/ingest_trader_qa.py` тАФ uses `pymilvus` with Zilliz's built-in embedding function

```
1. Load each markdown file from /docs/trader-qa/
2. Parse into Q&A chunks using regex on **Q\d+.** pattern
3. Extract metadata (theme_number, theme_name, source_file, doc_section)
4. Connect to Zilliz Cloud via MilvusClient(uri=ZILLIZ_URI, token=ZILLIZ_API_KEY)
5. Create collection with Function(BGELM, input="text_content", output="embedding")
6. Batch-insert raw text + metadata тАФ Zilliz embeds text_content server-side
7. Log chunk count and confirm index build
```

> No `openai` or `sentence-transformers` import needed for ingestion тАФ `pymilvus` only.

## 4. Backend API Route

### 4.1 Route: `POST /api/chat`

File: `app/api/chat/route.ts`

**Request body:**
```json
{
  "message": "How do I use covered calls to generate income on my NVDA position?",
  "trader_filter": null
}
```

**Response body:**
```json
{
  "answer": "...",
  "sources": [
    {
      "text_preview": "...",
      "source_file": "doc-1-options-and-swing-trading.md",
      "theme_name": "Options Income Strategies: Covered Calls, Puts, and the Wheel",
      "score": 0.91
    }
  ]
}
```

### 4.2 Pipeline Steps (inside the route handler)

Uses `@zilliz/milvus2-sdk-node` (Node.js SDK) тАФ no Python bridge needed from Next.js.

```
Step 1 тАФ Vector search in Zilliz Cloud  (built-in embedding handles query embedding)
  MilvusClient.search({
    collection_name: "trader_qa",
    data: [message],           тЖР raw text string, NOT a pre-computed vector
    anns_field: "embedding",
    limit: 5,
    output_fields: ["text_content", "question_text", "source_file", "theme_name"],
    filter: doc_section == "..." (optional, if trader_filter set)
  })
  Zilliz embeds the query string server-side using the same built-in model as ingestion
  тЖТ returns top-5 chunks with distance scores

Step 2 тАФ Build context string
  Concatenate top-k text_content chunks
  Format: "[Theme: {theme_name}]\n{text_content}\n---\n"

Step 3 тАФ LLM call
  POST https://api.openai.com/v1/chat/completions
  model: gpt-4o
  messages:
    system: (see prompt template below)
    user: message

Step 4 тАФ Return response + sources to frontend
```

> **Key optimization:** Because Zilliz's built-in pipeline embeds both the corpus at ingestion AND the query at search time, Step 1 of the original 5-step pipeline (separate OpenAI embedding call per query) is eliminated. Every user query costs one fewer API call.

### 4.3 System Prompt Template

```
You are a trading education assistant grounded in the Trader Q&A knowledge base.

The knowledge base features two trader perspectives:
- Tactical Opportunist (T1): options-heavy, short-dated, narrative-driven speculator
- Structured Growth Investor (T2): equity-primary, long-hold, quality-biased thematic investor

Use only the context below to answer. When both trader perspectives appear in the context,
present both views clearly labeled. If the context doesn't cover the question, say so.

Context:
{context}

Question: {question}
```
